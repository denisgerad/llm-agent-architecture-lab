Reducing Iteration Cycles

Every extra cycle with an agent costs context, time, and sometimes introduces new bugs while fixing old ones. These rules cut cycles at the source.

1 One problem per prompt
Agents given multiple issues in one prompt will fix some, partially fix others, and silently skip the rest. You then can't tell what changed or why.
✗ Multiple issues — guaranteed partial fix
"Fix the login bug, also the payment form isn't submitting, and can you rename the auth hook too"
✓ One issue — complete fix
"Only task: fix the login redirect after Supabase auth. It should go to /dashboard but goes to /. File: src/components/Auth/LoginForm.tsx"
2 Name the exact file and line
Agents search for context when you don't give it. They may find the wrong file, the wrong function, or infer incorrectly. Your blueprint makes this easy — you know every filename.
✗ Forces agent to guess location
"The subscription check isn't working"
✓ Zero ambiguity
"In src/hooks/useSubscription.ts, the isActive() check returns true even for expired plans. Expected: false when subscription.endDate < today"
3 State expected vs actual
The single biggest source of extra cycles is the agent not knowing what "correct" looks like. Always give both sides — what you got and what you expected.
✗ Agent guesses what "working" means
"The Razorpay modal isn't working correctly"
✓ Agent knows exactly what to achieve
"Expected: Razorpay modal opens on button click.
Actual: Uncaught TypeError: window.Razorpay is not a constructor
File: src/components/Subscription/PaymentForm.tsx, line 34"
4 Paste the actual error — all of it
Paraphrasing errors loses the exact message, stack trace, and error code the agent needs. Copy the full console output including the stack trace — the bottom lines are often more useful than the top.
✗ Paraphrased — agent works blind
"Getting a 401 error when calling the API"
✓ Full error — agent diagnoses precisely
POST https://xyz.supabase.co/rest/v1/subscriptions 401
{"code":"PGRST301","message":"JWT expired","hint":"Refresh session"}
→ Agent immediately knows: session refresh issue, not a query bug
5 Confirm before proceeding
After each agent fix, verify it works before giving the next task. Stacking unverified fixes means when something breaks you don't know which fix caused it — and the agent starts guessing.
✗ Stacked tasks — debug nightmare
"Also fix X, Y, and Z while you're at it" → three fixes applied → one breaks something → you don't know which
✓ Verify, then proceed
Fix 1 → test → ✓ confirmed → Fix 2 → test → ✓ confirmed
Clean git commit per fix. Bisect is trivial if anything regresses.
6 Give the agent scope boundaries
Without scope constraints, agents refactor things you didn't ask about, rename variables "for clarity," or restructure files. Explicitly say what is off-limits.
✗ No scope — agent rewrites half the file
"Fix the validation in the payment form"
✓ Scoped — agent touches only what's needed
"Fix only the card number validation in PaymentForm.tsx. Do not change any other validation, do not rename anything, do not refactor the component structure."
Code vs Environment

The most expensive mistake in AI-assisted development: reporting an environment issue as a code bug. The agent iterates on code that was never the problem. Learn to diagnose first, then report.

🌍 Environment Issues
◆Works in Chrome, fails in Safari — WebKit CSS/API difference
◆Works locally, fails in production — env vars, CORS, SSL, CDN headers
◆Works on your machine, fails for colleague — Node version, OS, local config
◆Works with fast internet, fails on 3G — timeout, race condition, load order
◆Works logged in, fails in incognito — cookies, storage, session state
◆Razorpay loads in dev, not staging — CSP, domain whitelist, test vs live keys
◆Supabase query works in dashboard, fails in app — RLS policy, JWT claims, anon vs auth role
💻 Code Issues
◆Fails in ALL browsers the same way
◆Reproducible every time with same steps
◆Error points to a specific file + line number
◆TypeScript/ESLint flags it at build time
◆Fails for all users equally
◆Unit test catches it in isolation
◆Same behaviour regardless of network/device
Diagnosis Flow — Run This Before Reporting Anything
Q
QUESTION TO ASK
IF YES → / IF NO →
1
REPRODUCE
Does it happen every time with the same steps?
YES → likely code issue, proceed
NO → likely race condition, timing, or network. Check loading states + async handling
2
CROSS-BROWSER
Does it happen in Chrome AND Firefox AND Safari?
ALL fail → code issue. Report to agent with full error
ONE browser → environment/compatibility issue. Tell agent: "Safari only — CSS/API compat issue, not logic bug"
3
LOCAL VS PROD
Does it work locally but fail in production/staging?
Check: env vars set? CORS configured? Supabase URL correct? Razorpay domain whitelisted? Don't ask agent to change code yet
4
AUTH STATE
Does it work logged out / in incognito but fail when logged in?
Supabase RLS issue or JWT problem — check Supabase logs first, not your code
Fails both ways → auth-unrelated code bug
5
ISOLATE
Can you reproduce with a minimal test? (comment out code, hardcode values)
YES → you've found the exact cause. Report that specific thing
Disappears when isolated → dependency, ordering, or context issue — report the interaction, not the symptom
6
NETWORK TAB
Check DevTools Network — what does the actual request/response look like?
4xx response → server/config issue (check Supabase logs, API keys)
Request never sent → code didn't reach the call. That's the bug location
📋 Safari / Cross-Browser Issue Checklist — Complete Before Asking Agent
Confirm: fails in Safari, works in Chrome. It's a compatibility issue, not a code bug
Check: does the feature use Date.parse()? Safari parses dates differently — use ISO 8601 format only
Check: any CSS gap in flexbox? Safari <14 doesn't support it. Use margin fallback
Check: any position: sticky? Safari requires -webkit-sticky prefix
Check: Razorpay script loading — Safari blocks third-party scripts differently. Check CSP headers
Check: any Clipboard API, Web Crypto, or IntersectionObserver? Check MDN Safari compat before asking agent to "fix the code"
Then report to agent: "Safari-only issue. Likely [specific cause]. Change only the compatibility fix, not the logic"
Dos & Don'ts

With a solid design and blueprint already in place, these are the disciplines that keep the project on track through implementation.

✓ Do
✓
Commit after every working feature Small, frequent commits. If an agent fix breaks something, you can revert to exactly before the bad change. Never let more than one agent session go uncommitted.
→ Prevents the "I don't know what changed" spiral
✓
Reference your blueprint filenames always Your Stage 2.5 blueprint has exact paths. Use them verbatim. "src/hooks/useAuth.ts" not "the auth hook file."
→ Eliminates file location ambiguity in long sessions
✓
Use LLM for decisions, agent for execution If you're unsure HOW to implement something, ask the LLM first. Get a plan. Then hand it to the agent to execute. Don't ask the agent to design while building.
→ Agents make permanent choices; LLMs explore options
✓
Run lint before and after every agent session npm run lint catches regressions the agent introduced silently. ESLint and your nesting rules will flag things faster than manual review.
→ Your Stage 2 config pays off here
✓
Tell the agent which module you're in "We are working in the Payment Module. Only touch files under src/components/Subscription/ and src/hooks/usePayment.ts."
→ Your module boundaries from Stage 2.5 are guardrails
✓
Check Supabase logs before code logs When a Supabase query fails, open the Supabase dashboard logs first. The error is usually there — JWT, RLS, or malformed query — before blaming your code.
→ Saves 2-3 agent cycles on every DB issue
✗ Don't
✗
Don't paste raw browser error messages alone "TypeError: Cannot read properties of undefined" means nothing without the file, line, and what you were doing. Always add context before the error.
→ Raw errors alone cause agents to guess — and guess wrong
✗
Don't let the agent change your folder structure Your Stage 2.5 blueprint is intentional. If the agent suggests moving files or renaming folders, ask it to work within the existing structure instead.
→ Blueprint drift breaks RAG, imports, and team alignment
✗
Don't ignore TypeScript errors "for now" Suppressing TS errors with // @ts-ignore hides real problems. Razorpay and Supabase types exist precisely to catch mistakes at compile time, not runtime.
→ Payment bugs with type suppressions are production bugs
✗
Don't ask "why isn't this working" without diagnosis Run the diagnosis flow first (Code vs Environment section). Report what you found, not just the symptom. Agents given symptoms iterate; agents given causes fix.
→ Symptoms generate hypotheses; causes generate solutions
✗
Don't accumulate unreviewed agent output Read every file the agent creates or modifies. Agents introduce subtle bugs — a hardcoded value, a missing error boundary, an auth check that's always true. You own the code.
→ Agent output is a first draft, not a final answer
✗
Don't start a new feature on a broken state If there's a known bug or lint error, fix it first. Building on broken ground means the agent's new work inherits the brokenness — and the root cause gets harder to find.
→ Technical debt compounds fast in agent-assisted code
